# PATTERN — 3D

Extends DESIGN.md. Works alongside any STYLES/ file. Never overrides CONFIG.md or METHOD.md.

---

## Core Principle

Depth, perspective, and three-dimensional space as visual design elements. 3D can be CSS-only (transforms, perspective) for subtle depth effects, or WebGL-based (Three.js, React Three Fiber) for full 3D scenes. Choose the minimum necessary approach.

---

## When to Use

- Product visualization (showing an object from multiple angles)
- Hero sections where spatial depth differentiates from flat competitors
- Interactive data visualization with spatial dimensions
- Portfolio or creative work where 3D is itself the medium
- Ambient backgrounds that add depth without competing with content

Do not use on: content-heavy pages, functional UIs, dashboards, or anywhere performance headroom is limited.

---

## Approach 1: CSS 3D (Lightweight, No Dependencies)

CSS `perspective` and `transform` create genuine 3D space for DOM elements.

### Setup

```css
.scene {
  perspective: 1000px;
  perspective-origin: 50% 50%;
}

.card-3d {
  transform-style: preserve-3d;
  transition: transform 600ms ease-out;
}

.card-3d:hover {
  transform: rotateY(8deg) rotateX(4deg);
}
```

### Tilt-on-hover (pointer-tracking)

```tsx
const useTilt = (maxTilt = 10) => {
  const ref = useRef<HTMLDivElement>(null)

  const handleMouseMove = (e: MouseEvent) => {
    const el = ref.current
    if (!el) return
    const rect = el.getBoundingClientRect()
    const x = (e.clientX - rect.left) / rect.width - 0.5   // -0.5 to 0.5
    const y = (e.clientY - rect.top) / rect.height - 0.5

    el.style.transform = `
      perspective(800px)
      rotateX(${-y * maxTilt}deg)
      rotateY(${x * maxTilt}deg)
    `
  }

  const handleMouseLeave = () => {
    if (ref.current) ref.current.style.transform = ""
  }

  return { ref, onMouseMove: handleMouseMove, onMouseLeave: handleMouseLeave }
}
```

Pair with `transition: transform 200ms ease-out` for enter, and a slower transition on leave (`600ms ease-out`) for a natural settle.

### Card flip

```tsx
function FlipCard({ front, back }) {
  const [flipped, setFlipped] = useState(false)
  return (
    <div
      className="relative h-64 w-48 cursor-pointer [perspective:1000px]"
      onClick={() => setFlipped(f => !f)}
    >
      <div
        className={cn(
          "relative h-full w-full duration-700 [transform-style:preserve-3d]",
          flipped && "[transform:rotateY(180deg)]"
        )}
        style={{ transition: "transform 600ms ease-in-out" }}
      >
        <div className="absolute inset-0 [backface-visibility:hidden]">{front}</div>
        <div className="absolute inset-0 [backface-visibility:hidden] [transform:rotateY(180deg)]">{back}</div>
      </div>
    </div>
  )
}
```

## Approach 2: React Three Fiber (Full 3D)

For genuine 3D objects, scenes, and WebGL rendering.

### Minimal setup

```tsx
import { Canvas } from "@react-three/fiber"
import { OrbitControls, Environment } from "@react-three/drei"

function Scene() {
  return (
    <Canvas
      camera={{ position: [0, 0, 5], fov: 45 }}
      gl={{ antialias: true, alpha: true }}
      style={{ position: "absolute", inset: 0 }}
    >
      <ambientLight intensity={0.5} />
      <directionalLight position={[10, 10, 5]} intensity={1} />
      <mesh>
        <boxGeometry args={[1, 1, 1]} />
        <meshStandardMaterial color="white" roughness={0.3} metalness={0.1} />
      </mesh>
      <Environment preset="city" />
      <OrbitControls enableZoom={false} />
    </Canvas>
  )
}

function HeroWithCanvas() {
  return (
    <section className="relative h-screen overflow-hidden">
      <Scene />
      <div className="relative z-10 flex h-full items-center justify-center">
        {/* content overlaid on 3D scene */}
      </div>
    </section>
  )
}
```

### Model loading

```tsx
import { useGLTF } from "@react-three/drei"

function Model({ url }: { url: string }) {
  const { scene } = useGLTF(url)
  return <primitive object={scene} />
}

// Preload at module level
useGLTF.preload("/models/product.glb")
```

### Scroll-linked 3D rotation (R3F + scroll)

```tsx
import { useScroll } from "@react-three/drei"
import { useFrame } from "@react-three/fiber"

function RotatingModel() {
  const scroll = useScroll()
  const meshRef = useRef()

  useFrame(() => {
    meshRef.current.rotation.y = scroll.offset * Math.PI * 2
  })

  return <mesh ref={meshRef}>{/* ... */}</mesh>
}
```

## Approach 3: CSS-Only Depth (No Transform)

When full 3D is too heavy but depth is desired — using blur, scale, and opacity to simulate z-depth:

```css
/* Closer elements: crisp, full-size, full opacity */
.layer-front  { transform: scale(1);    opacity: 1;   filter: blur(0); }
/* Mid elements: slightly smaller, slightly transparent */
.layer-mid    { transform: scale(0.95); opacity: 0.8; filter: blur(0.5px); }
/* Background: smaller, transparent, blurred */
.layer-back   { transform: scale(0.85); opacity: 0.5; filter: blur(2px); }
```

## Performance Constraints

- Three.js/R3F: always use `Suspense` with a fallback — GLTF loading blocks rendering
- Limit active draw calls — prefer merged geometries and instancing for repeated objects
- Use `draco` compression for GLTF models (typically 70-90% size reduction)
- Disable antialiasing on mobile (`gl={{ antialias: false }}` at breakpoint) — significant perf gain
- CSS tilt effects: throttle `mousemove` handler with `requestAnimationFrame`, not inline
- WebGL canvas should never cover the full page on low-end devices — detect and degrade

---

## Accessibility

- 3D canvas content is invisible to screen readers — all semantic content must exist in the DOM alongside the canvas
- Motion in 3D scenes must respect `prefers-reduced-motion` — disable auto-rotation, reduce or freeze animation
- Never make critical information only available through 3D interaction
- Provide a non-3D fallback for environments where WebGL is unavailable

```tsx
const supportsWebGL = () => {
  try {
    return !!document.createElement("canvas").getContext("webgl2")
  } catch { return false }
}

function Hero() {
  if (!supportsWebGL()) return <StaticHero />
  return <HeroWithCanvas />
}
```

---

## Fallback Requirements

- CSS 3D: `transform-style: preserve-3d` is widely supported — no fallback needed beyond `prefers-reduced-motion`
- WebGL: provide a static image or flat component for unsupported environments
- SSR: R3F components must be dynamically imported with `ssr: false` in Next.js

```tsx
const Scene = dynamic(() => import("./Scene"), { ssr: false })
```

---

## Common Pitfalls

- **Z-fighting:** two surfaces at the same z-depth flickering against each other. Fix by slightly separating their z positions.
- **Performance on mobile:** full 3D scenes are often not viable on mid-range phones. Always test on real hardware.
- **Tilt overdone:** `maxTilt > 15deg` starts looking like a bug. Keep it subtle — 8–12deg max.
- **Content unreadable over 3D:** text overlaid on a busy 3D scene needs a scrim or backdrop. Never rely on the 3D background having sufficient contrast.
- **Forgetting SSR:** R3F will crash during server rendering. Dynamic import is mandatory.

## Approach 4: Spline (No-Code, Fastest Path)

Spline is a browser-based visual 3D editor — the closest analogue to Figma for 3D. It exports scenes that embed directly into web projects.

### Strengths
- Fastest path from idea to embedded 3D: polished hero animation in under 2 hours
- Built-in physics, interactive states, and scroll-linked animations — no Three.js knowledge needed
- Native Webflow integration (drag-and-drop embed)
- Real-time collaboration

### Weaknesses
- Closed/proprietary: no open-source fallback if Spline changes pricing or API
- Performance is not user-controllable: the exported `scene.splinecode` + runtime bundle is a black box — typically 2–8 MB
- CPU execution: even simple Spline scenes can run 17+ seconds of CPU on desktop; on mobile, GPU thermal throttling and battery drain are real
- No tree-shaking: the full Spline runtime ships regardless of scene complexity

### React / Next.js lazy-loading pattern

Never import `@splinetool/react-spline` at the top of a page file. Always lazy-load and defer until the section is visible:

```tsx
import dynamic from "next/dynamic"
import { useRef, useEffect, useState } from "react"

const Spline = dynamic(() => import("@splinetool/react-spline"), {
  ssr: false,
  loading: () => (
    <div className="absolute inset-0 animate-pulse bg-muted rounded-xl" aria-hidden="true" />
  ),
})

function SplineHero({ sceneUrl }: { sceneUrl: string }) {
  const containerRef = useRef<HTMLDivElement>(null)
  const [shouldLoad, setShouldLoad] = useState(false)

  useEffect(() => {
    if (!containerRef.current) return
    const observer = new IntersectionObserver(
      ([entry]) => {
        if (entry.isIntersecting) {
          setShouldLoad(true)
          observer.disconnect()
        }
      },
      { threshold: 0.1 }
    )
    observer.observe(containerRef.current)
    return () => observer.disconnect()
  }, [])

  return (
    <div ref={containerRef} className="relative w-full h-screen overflow-hidden">
      {shouldLoad && (
        <Spline
          scene={sceneUrl}
          className="absolute inset-0 w-full h-full"
        />
      )}
    </div>
  )
}
```

### Spline performance checklist
1. Use "Export → Code" (not the viewer iframe embed) for React/Next.js — smaller runtime
2. Reduce polygon count in the Spline editor: fewer materials, fewer textures
3. Lazy load via IntersectionObserver (pattern above) — never initialise on page load
4. Provide a static image or looping video fallback when the scene is decorative
5. Test LCP and INP with Lighthouse before shipping — a Spline scene can delay FCP by 2–4 seconds on 3G

## Tool Decision Matrix

| Situation | Use |
|---|---|
| Results needed in < 2 hours, no code | **Spline** |
| Webflow project + visual editor preferred | **Spline** (native integration) |
| React project, full control, team knows JS | **React Three Fiber** |
| Non-React project, maximum WebGL control | **Vanilla Three.js** |
| Assets created in Blender, exported to web | Blender → export `.glb` → load with **R3F** or **Three.js** |
| Hero animation, purely decorative | Consider 5-second looping **video** — same narrative value at 1/10 the complexity |
| Content-first page (blog, docs, e-commerce) | **None** — performance cost not justified |

### Honest cost of 3D

Before committing to WebGL: a 2–8 MB JavaScript payload delays FCP and TTI by 2–4 seconds on 3G. A simple Spline scene can consume 17+ seconds of CPU on desktop. Mid-range Android GPU performance varies 10–20× from flagship devices — what runs at 60fps on a Pixel 9 may jank at 20fps on a budget phone. The decision test: **could a 5-second looping video serve the same narrative purpose at 1/10 the complexity?** If yes, use video.

## Expanded Performance Patterns

### IntersectionObserver lazy initialization (vanilla Three.js)

Never initialise a Three.js scene on page load if the canvas is below the fold:

```tsx
function useLazyThree(initFn: () => (() => void) | void) {
  const containerRef = useRef<HTMLDivElement>(null)

  useEffect(() => {
    const el = containerRef.current
    if (!el) return

    let cleanup: (() => void) | void

    const observer = new IntersectionObserver(
      ([entry]) => {
        if (entry.isIntersecting) {
          cleanup = initFn()
          observer.disconnect()
        }
      },
      { threshold: 0.1 }
    )
    observer.observe(el)

    return () => {
      observer.disconnect()
      cleanup?.()
    }
  }, [initFn])

  return containerRef
}

// Usage
function ThreeSection() {
  const initScene = useCallback(() => {
    // Three.js setup here — runs only when section enters viewport
    const scene = new THREE.Scene()
    // ...
    const rafId = requestAnimationFrame(animate)
    return () => {
      cancelAnimationFrame(rafId)
      renderer.dispose()
    }
  }, [])

  const containerRef = useLazyThree(initScene)

  return <div ref={containerRef} className="w-full h-96" aria-hidden="true" />
}
```

### Model compression workflow

Three.js models should be processed before deployment:

```bash
# Install gltf-transform CLI
npm install -g @gltf-transform/cli

# Optimize: compress geometry, resize textures, deduplicate
gltf-transform optimize input.glb output.glb --texture-compress webp

# Apply Draco compression for geometry (most impactful for dense meshes)
gltf-transform draco input.glb output.glb
```

Load Draco-compressed models in R3F:

```tsx
import { useGLTF } from "@react-three/drei"
import { DRACOLoader } from "three/examples/jsm/loaders/DRACOLoader"
import { GLTFLoader } from "three/examples/jsm/loaders/GLTFLoader"
import { useLoader } from "@react-three/fiber"

// Configure Draco loader globally (once, in your scene setup)
useGLTF.setDecoderPath("https://www.gstatic.com/draco/versioned/decoders/1.5.6/")

function CompressedModel({ url }: { url: string }) {
  const { scene } = useGLTF(url)
  return <primitive object={scene} />
}

// Preload at module level — avoids waterfall on first render
useGLTF.preload("/models/product.glb")
```

### Texture size limits

- 512×512 is sufficient for most web hero elements — the canvas rarely renders at native texture resolution
- 1024×1024 max for close-up product configurators
- Use WebP format (`gltf-transform` supports `--texture-compress webp`)
- Dispose textures explicitly when scenes tear down:

```tsx
useEffect(() => {
  return () => {
    // Prevent GPU memory leaks
    scene.traverse((obj) => {
      if (obj instanceof THREE.Mesh) {
        obj.geometry.dispose()
        if (Array.isArray(obj.material)) {
          obj.material.forEach((m) => {
            m.map?.dispose()
            m.dispose()
          })
        } else {
          obj.material.map?.dispose()
          obj.material.dispose()
        }
      }
    })
    renderer.dispose()
  }
}, [])
```

### GPU acceleration CSS hints

Apply to the canvas container — promotes it to a compositor layer before animation begins:

```css
.three-canvas-container {
  transform: translateZ(0);
  will-change: transform; /* apply only while animating; remove after */
  contain: layout style paint;
}
```

### R3F with React Suspense for async model loading

```tsx
import { Suspense } from "react"
import { Canvas } from "@react-three/fiber"
import { useGLTF, Environment } from "@react-three/drei"

function Model() {
  const { scene } = useGLTF("/models/product.glb")
  return <primitive object={scene} />
}

function ProductScene() {
  return (
    <Canvas camera={{ position: [0, 0, 5], fov: 45 }} gl={{ antialias: true }}>
      <ambientLight intensity={0.4} />
      <directionalLight position={[5, 8, 3]} intensity={1.2} castShadow />
      <Suspense fallback={null}>
        {/* Model renders only after GLTF is loaded; Suspense shows nothing meanwhile */}
        <Model />
        <Environment preset="studio" />
      </Suspense>
    </Canvas>
  )
}

// In the page — wrap entire canvas in Suspense with a visual fallback
function ProductHero() {
  if (!supportsWebGL()) return <StaticProductImage />

  return (
    <section className="relative h-screen overflow-hidden">
      <Suspense
        fallback={
          <div className="absolute inset-0 flex items-center justify-center bg-muted animate-pulse" />
        }
      >
        <ProductScene />
      </Suspense>
      <div className="relative z-10 flex h-full items-center">
        {/* Content overlaid on the 3D scene */}
      </div>
    </section>
  )
}
